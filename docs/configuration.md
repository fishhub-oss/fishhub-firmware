# Configuration

## Provisioning a new device (primary method)

Device credentials are stored in ESP32 NVS (non-volatile storage) and written via a captive-portal Wi-Fi AP on first boot. This is the intended flow for all deployed devices.

### Prerequisites

- FishHub backend running and reachable from the device's Wi-Fi network
- A provisioning code generated by the FishHub web app (issued via `POST /api/devices/provision` on the server)

### Steps

1. Flash the firmware:
   ```bash
   pio run --target upload
   ```
2. On first boot, `isProvisioned()` returns false and the device starts the AP **"FishHub-Setup"**.
3. From a phone or laptop, connect to **FishHub-Setup** (no password).
4. Open `http://192.168.4.1` in a browser. The captive portal form shows:
   - **Wi-Fi network** — dropdown populated from a background scan, with a manual entry fallback
   - **Password** — Wi-Fi password (show/hide toggle)
   - **Provisioning code** — one-time code from the web app
5. Submit the form. The device:
   1. Connects to the specified Wi-Fi network.
   2. POSTs `{"code":"<provisioning-code>"}` to `SERVER_URL/devices/activate`.
   3. Receives `202 Accepted` with a device JWT and device ID.
   4. Polls `GET /devices/{id}/status` (every 2 s, up to 60 s) until the server reports MQTT credentials are ready.
   5. Writes all credentials to NVS atomically (the `provisioned` flag is set last).
   6. Reboots into normal operation.

If any step fails (wrong code, Wi-Fi unreachable, server error, polling timeout) the portal restarts and shows an error message. The device can be retried without reflashing.

### NVS keys written during provisioning

| Key | Written by | Description |
|---|---|---|
| `wifi_ssid` | Captive portal | Wi-Fi network name |
| `wifi_pass` | Captive portal | Wi-Fi password |
| `device_id` | Activation response | UUID of the device |
| `device_jwt` | Activation response | RS256-signed JWT used as Bearer token for HTTP and as MQTT password |
| `mqtt_username` | Status poll response | HiveMQ username |
| `mqtt_password` | Status poll response | HiveMQ password |
| `mqtt_host` | Status poll response | HiveMQ broker hostname |
| `provisioned` | Written last | Atomic commit flag — device is only considered provisioned when this is `"1"` |

`SERVER_URL` is always read from `config.h` at compile time — it is never stored in NVS.

### Reconfiguration (update Wi-Fi credentials)

Hold the **BOOT button** (GPIO 0) for **3 seconds** while the device is running. The device restarts the AP in reconfiguration mode — the form shows only the Wi-Fi fields (no provisioning code).

On submit, the new Wi-Fi credentials are written to temporary NVS keys first. If the connection test succeeds, they are promoted to the real keys and the device reboots. If it fails, the temporary keys are deleted and the portal shows an error — **existing MQTT credentials are never overwritten on failure**.

### Factory reset

Hold the **BOOT button** for **10 seconds**. The device calls `nvsStore.clear()` (erases all NVS keys including `provisioned`) and reboots into fresh provisioning mode.

## `config.h` (development fallback)

`include/config.h` is **gitignored** and acts as a compile-time fallback when NVS values are absent. It is useful during local development when provisioning via captive portal is inconvenient.

Create it from the template:

```bash
cp include/config.h.example include/config.h
```

Then fill in the defines:

```cpp
#define WIFI_SSID      "your-ssid"
#define WIFI_PASSWORD  "your-password"
#define SERVER_URL     "http://192.168.x.x:8080"

// MQTT / device identity — populated by provisioning; set here for dev only
#define DEVICE_ID      ""
#define DEVICE_TOKEN   ""
#define MQTT_HOST      "your-broker.hivemq.cloud"
#define MQTT_PORT      8883
```

| Define | Description |
|---|---|
| `WIFI_SSID` | SSID of the Wi-Fi network the ESP32 will join (fallback when NVS is empty) |
| `WIFI_PASSWORD` | Wi-Fi password (fallback when NVS is empty) |
| `SERVER_URL` | Base URL of the FishHub backend — **required at compile time**, never stored in NVS. Use the local IP, not `localhost`. |
| `DEVICE_ID` | Device UUID — leave empty; populated by provisioning |
| `DEVICE_TOKEN` | Unused in current firmware; leave empty |
| `MQTT_HOST` | HiveMQ broker hostname (fallback when NVS is empty) |
| `MQTT_PORT` | HiveMQ broker port (default `8883`) |

On a provisioned device the NVS values always take precedence over `config.h` defines.

## `include/pins.h`

```cpp
#define ONE_WIRE_PIN      4   // DS18B20 data line
#define RESET_BUTTON_PIN  0   // BOOT button (active LOW) — 3 s reconfigure, 10 s factory reset
#define RELAY_LIGHT_PIN   X   // relay for the light actuator
```

Change `ONE_WIRE_PIN` or `RELAY_LIGHT_PIN` if you wire peripherals to different GPIO pins.
